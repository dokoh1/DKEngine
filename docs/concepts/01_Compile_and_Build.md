# 컴파일(Compile) vs 빌드(Build)

흔히 섞어 쓰지만 둘은 **범위가 다르다.**

> **컴파일 = 소스 파일 하나를 기계어 조각(.obj)으로 바꾸는 "한 단계"**
> **빌드 = 소스들을 실행 가능한 최종 결과물(.exe)로 만드는 "전체 과정"**

즉 **컴파일은 빌드의 일부**다. 빌드는 컴파일 + 링크 + 그 외 잡일(리소스 처리, 파일 복사 등)을
**순서대로 조율(orchestration)** 하는 더 큰 개념이다.

---

## 1. 큰 그림: 소스에서 실행파일까지

```
   [전처리]        [컴파일]          [링크]
main.cpp ─► (TU) ─► main.obj ─┐
Window.cpp ─► ... ─► Window.obj ┤
App.cpp ───► ... ─► App.obj ───┼─► link.exe ─► DKEngine.exe
                                │
              user32.lib / d3d12.lib(라이브러리) ─┘

└──────────────── 이 전체가 "빌드" ────────────────┘
        └── 파일 하나씩의 "컴파일" ──┘
```

- 컴파일은 **파일 단위**로 일어난다(`main.cpp` 따로, `Window.cpp` 따로).
- 링크는 그 조각들을 **한 번에** 모아 하나의 실행파일로 합친다.
- 이 모든 단계를 자동으로 돌리는 주체가 **빌드 시스템**(우리 경우 MSBuild).

---

## 2. 단계별 자세히

### 단계 0. 전처리 (Preprocessing)
컴파일러가 본격적으로 일하기 전, **텍스트 치환**을 먼저 한다.
- `#include <Windows.h>` → 그 헤더 내용을 **그 자리에 통째로 붙여넣음**
- `#define`, `#ifdef` 같은 매크로 처리, 주석 제거
- 결과물: 한 `.cpp` + 거기 포함된 모든 헤더가 합쳐진 거대한 텍스트 한 덩어리
  → 이걸 **번역 단위(Translation Unit, TU)** 라고 부른다.

> 그래서 헤더에 무언가 추가하면, 그 헤더를 include한 **모든 .cpp가 다시 컴파일**된다.
> (헤더 변경이 빌드를 느리게 만드는 이유)

### 단계 1. 컴파일 (Compile) — 좁은 의미의 "컴파일"
번역 단위(TU)를 **기계어로 번역**한다. 내부적으로는:
- 문법 검사(syntax) → 타입 검사(semantic) → 최적화 → 어셈블리 → 기계어
- 결과물: **목적 파일(object file, `.obj`)**. CPU 명령어 조각이 담긴 바이너리.

여기서 나는 오류가 **컴파일 에러**다 (MSVC에서 `C2065`, `C2660` 같은 `C####` 코드):
- 세미콜론 빠짐, 오타, 타입 불일치, 선언 안 된 변수 사용 등 **한 파일 안에서** 잡히는 문제.

**핵심**: 컴파일 시점엔 다른 파일을 모른다. `Window.cpp`를 컴파일할 때
`Application`의 함수가 "어딘가 있다"는 **선언(헤더)** 만 보면 통과한다.
실제 그 함수의 **본체가 진짜 존재하는지**는 따지지 않는다 → 그건 링크의 몫.

### 단계 2. 링크 (Link)
흩어진 `.obj`들과 **라이브러리(`.lib`)** 를 모아 하나의 실행파일로 합친다.
- 각 `.obj`는 "나는 `Application::Run`을 호출하는데, 그 실체는 다른 곳에 있다"는
  **미해결 참조(symbol)** 를 갖고 있다.
- 링커는 모든 `.obj`/`.lib`를 훑어 **"그 실체가 여기 있다"를 찾아 연결**한다(심볼 해소).
- 결과물: 실행 가능한 `DKEngine.exe`.

여기서 나는 오류가 **링크 에러** (MSVC에서 `LNK2019`, `LNK2001` 같은 `LNK####` 코드):
- "unresolved external symbol" = **선언은 있는데 구현(본체)을 못 찾음**.
- 대표 원인: 함수를 선언만 하고 `.cpp`에 정의 안 함 / 필요한 **라이브러리를 링크 안 함**.

> **Phase 2 예고**: D3D12 함수를 호출하는데 `d3d12.lib`/`dxgi.lib`를 링크에 추가하지 않으면,
> 컴파일은 멀쩡히 되는데 **링크에서 `LNK2019`** 가 터진다. 그때 이 문서를 떠올리면 된다.

### 단계 3. 그 외 빌드 작업
빌드는 컴파일·링크 외에도 빌드 시스템이 챙기는 일들을 포함한다:
- HLSL 셰이더(`.hlsl`) 컴파일 (Phase 3부터)
- 리소스/에셋 파일을 출력 폴더(`build/`)로 복사
- 빌드 전/후 사용자 지정 단계(post-build step) 실행

## 3. 컴파일 에러 vs 링크 에러 (실전에서 가장 유용)

| | 컴파일 에러 | 링크 에러 |
|---|------------|-----------|
| 언제 | 파일 하나를 번역할 때 | 모든 조각을 합칠 때 |
| MSVC 코드 | `C2xxx` 등 | `LNK2xxx` 등 |
| 전형적 원인 | 오타, 타입 불일치, 세미콜론, 미선언 | 정의 누락, **lib 링크 누락**, 중복 정의 |
| 비유 | 문장의 **문법**이 틀림 | 문법은 맞는데 **참조한 대상이 실제로 없음** |

이 구분을 알면 빌드 실패 메시지를 보고 **"어느 단계에서 깨졌는지"** 를 즉시 알 수 있다.

---

## 4. 왜 컴파일과 링크를 굳이 나눴을까?
한 번에 .exe를 만들지 않고 단계를 쪼갠 데는 이유가 있다.

1. **증분 빌드(incremental build)**: `Window.cpp` 한 개만 고치면, 그 파일만 다시 컴파일하고
   나머지 `.obj`는 재사용해 링크만 다시 한다 → 빌드가 훨씬 빠르다.
2. **분리 컴파일(separate compilation)**: 파일별로 독립 컴파일되므로 큰 프로젝트를
   여러 CPU 코어가 **병렬로** 컴파일할 수 있다(.vcxproj의 `MultiProcessorCompilation`).
3. **라이브러리 재사용**: 미리 컴파일된 `.lib`/`.dll`을 링크만 하면 된다(소스 없이도 사용).
   Windows SDK의 `d3d12.lib`가 바로 그런 예.

---

## 5. 우리 프로젝트에 대입하면

| 명령/행동 | 무슨 일이 일어나나 |
|-----------|-------------------|
| VS에서 **빌드(Ctrl+Shift+B)** | MSBuild가 `.vcxproj`를 읽어 `main/Window/Application.cpp`를 각각 **컴파일** → `.obj` 생성 → `user32.lib` 등과 **링크** → `build\x64\Debug\DKEngine.exe` |
| **F5 (디버깅 시작)** | 위 빌드를 (필요 시) 수행한 뒤, 만들어진 `.exe`를 **디버거에 붙여 실행** |
| 한 파일만 수정 후 빌드 | 그 파일만 재컴파일(증분 빌드) |

---

## 6. 한 줄 요약
- **컴파일**: `.cpp` 한 개 → `.obj` 한 개 (파일 단위 번역). 좁은 의미.
- **링크**: 여러 `.obj` + `.lib` → 하나의 `.exe` (조각 합치기 + 심볼 연결).
- **빌드**: 전처리 → 컴파일 → 링크 → 기타 작업을 **빌드 시스템이 통째로 자동 수행**하는 전체 과정.
- 실패하면 **컴파일 에러(C####)** 인지 **링크 에러(LNK####)** 인지부터 확인하면 원인이 반쯤 보인다.

> 관련: MSVC 컴파일러·Windows SDK가 각 단계에서 하는 역할은
> [`docs/Phase00_Setup.md`](../Phase00_Setup.md)의 "2-1 보충" 참고.
